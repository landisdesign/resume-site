Java experience and philosophy

I started programming BASIC when I was 9, with a Apple IIe in a school computer room, and was programming in Assembly by the time I was 12. I always wanted to stretch the edge of what I could do with my computers. In high school and college I learned Pascal and C++ to write programs on Macintoshes. Eventually I learned Java because I wanted to know how JSPs worked.

Experience

Most of my Java experience has been on WebLogic and Tomcat, creating middleware code for americanfunds.com. Specific projects include:

  • Migrated to and maintained the homegrown Struts- and XML-based CMS (2004–2011)
  • Migrated the Literature Center to use this homegrown CMS instead of its own separate Struts instance (2006)
  • Wrote an entire content site conjoined with a DST financial transaction site, including the cross-server login status management (2009)
  • Creating components and templates in AEM, writing both the Java and (at the time) ExtJS elements of each (2012–2014)

These were done in addition to the front-end development that was my focus.

Back-end Development Philosophy

When I first read WROX’s Java Server Pages tome in 2003, I was struck by their reference to programming patterns. When I finally got my hands on Design Patterns, I was completely blown away.

Ever since that time I see my programs in terms of patterns. When I start seeing if/else trees or overly complex switch/case blocks, I immediately start considering applying Strategy patterns instead. When I see collections of functionality that are shared across multiple classes, I start considering Abstract Factories and Templates. Dependency injection is simply the next in a long line of useful ideas boiled down into a recognizable label and process.

For me, the second most influential book after Design Patterns is Steve McConnell’s Code Complete. His ideas of complexity management resonate strongly with me. Looking at variable lifespans, checking the complexity of a method by counting the number of conditional checks, making names as absolutely readable as possible, using white space as a formatting tool, all are techniques I use and appreciate on a daily basis.

Between the complexity and simplicity of both books, I look for a balance between inflexibility and gold-plating. When it is simple to write for the future, I will do that, but it must be simple. If it is unreadable by the next developer coming onto a project, it becomes far more costly in the future, as lava code accumulates that nobody wants to touch.

While I”ve been on my sabbatical, I”ve been looking at what’s been introduced since CG upgraded to Java SE 6. It’s great to see Oracle taking a stab at functional programming, and I’m curious to see how it is being adopted in the workplace.
